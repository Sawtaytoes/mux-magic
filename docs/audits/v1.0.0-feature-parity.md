## Partially ported

### 1. SmartMatchModal — Plex extra-type suffix selector missing

**Legacy file:** `public/builder/js/components/specials-mapping-modal.js` (introduced `a7fef431`, deleted `28534ec5`, re-ported by worker 58 Part B)

**React equivalent:** `packages/web/src/components/SmartMatchModal/SmartMatchModal.tsx`

**What the v1.0.0 version had:** Each row in the mapping table rendered a second control row (`[data-mapping-suffix-row]`) with a `<select>` dropdown listing nine Plex extra-type suffixes: `— no type —`, `Trailer` (`-trailer`), `Featurette` (`-featurette`), `Interview` (`-interview`), `Behind the Scenes` (`-behindthescenes`), `Scene` (`-scene`), `Deleted Scene` (`-deleted`), `Short` (`-short`), `Other` (`-other`). The suffix row was hidden until a candidate name was selected, then shown automatically. On Apply the final filename was `<base name> <suffix>` (e.g. `Commentary with Director -featurette`). Two helpers supported it: `extractSuffixFromStem` (read an existing Plex suffix back out of a filename so re-running re-selected the prior value) and `inferSuffixFromName` (auto-suggest a suffix from the candidate label by keyword matching on "trailer", "interview", etc.).

**What the React port has:** The modal is present and functional (worker 58 Part B) but the Plex-suffix dropdown is not ported. The feature is acknowledged as deferred in `packages/web/src/components/SmartMatchModal/SmartMatchModal.mdx` line 12: "The Plex-suffix selector remains deferred to a follow-up worker."

**User-visible impact:** A v1.0.0 user who relied on the suffix picker to produce correctly-typed Plex extras (`Commentary -featurette`, `Deleted Scenes -deleted`, etc.) must post-process filenames manually after Apply in the current React UI.

**Specific gaps in the React component:**
- `packages/web/src/components/SmartMatchModal/SmartMatchModal.tsx` has no `plexSuffix` field on the row state type
- `packages/web/src/components/SmartMatchModal/smartMatchTypes.ts` has no `PlexExtraType` enum or `suffix` field on `SmartMatchRow`
- No `extractSuffixFromStem` / `inferSuffixFromName` helpers

**Recommended follow-up worker:** `7a` — `smartmatch-plex-suffix-selector`

### 2. NsfRunResults — SmartMatch modal does not auto-open on step completion

**Legacy file:** `public/builder/js/run-sequence/step-results.js` (deleted `28534ec5`)

**React equivalent:** `packages/web/src/components/NsfRunResults/NsfRunResults.tsx`

**What the v1.0.0 version had:** After an NSF step completed with leftover unnamed files AND available DVDCompare candidates, the legacy `wireNameSpecialFeaturesResults` function automatically opened the Smart Match modal once per step run (`step._smartMatchAutoOpened` guard prevented double-opens on re-render). The user could not miss unmatched files — the modal appeared without any manual interaction.

**What the React port has:** The `NsfRunResults` component exposes a "✨ Fix Unnamed" button but does not auto-open the `SmartMatchModal`. The user must notice the button and click it. The `openSmartMatch` handler at `packages/web/src/components/NsfRunResults/NsfRunResults.tsx:46` is only wired to the button's `onClick`.

**User-visible impact:** A v1.0.0 user who relied on the auto-open to be notified about leftover files at run-completion will silently miss them in the current React UI unless they notice the "✨ Fix Unnamed" button appearance on the job card.

**Specific gaps in the React component:**
- `packages/web/src/components/NsfRunResults/NsfRunResults.tsx` — no `useEffect` that sets `smartMatchModalAtom` on initial mount / step completion event
- No `hasAutoOpened` guard (equivalent of legacy `step._smartMatchAutoOpened`)

**Recommended follow-up worker:** `7b` — `smartmatch-auto-open-on-nsf-completion`
