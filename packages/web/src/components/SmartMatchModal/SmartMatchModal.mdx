# SmartMatchModal

Batch "Fix Unnamed" / Smart Match table modal for the `nameSpecialFeaturesDvdCompareTmdb` (NSF) command. Opens after an NSF run completes with leftover unnamed files AND at least one DVDCompare candidate. Each row is a leftover file paired with the highest-confidence candidate from the run's `possibleNames`; the user reviews, optionally overrides via dropdown, then clicks **Apply** to fire a batch of `POST /files/rename` calls.

Worker 58 / Part B ported this modal from the v1.0.0 legacy `packages/web/public/builder/js/components/specials-mapping-modal.js` (added 2026-05-08 in `a7fef431`, deleted 2026-05-10 in `28534ec5` during the React conversion, never ported). Worker 6f then restored the ✏-custom-name affordance the React port had deferred — see "Per-row ✏ custom name" below. The Plex-suffix selector remains deferred to a follow-up worker.

## Behaviour

- **Open** → set `smartMatchModalAtom` to `{ jobId, stepId, sourcePath, suggestions }`. Worker 25 moved scoring server-side: `suggestions` is `FileSuggestion[]` with `rankedCandidates: ScoredCandidate[]` pre-baked, so the modal is a pure presenter.
- **Per-row include checkbox** → high-confidence rows (≥ `LOW_CONFIDENCE_THRESHOLD = 0.6`) are pre-checked; low-confidence rows are pre-unchecked so the user explicitly opts-in to a sketchy match.
- **Per-row override dropdown** → lists every candidate in `name — XX%` form, ordered by confidence. Picking a different option swaps the rename target for that row.
- **Per-row ✏ custom name** (worker 6f) → toggles the cell from the picker dropdown to a free-text input. The picker's selection and the typed value are kept as two independent fields (legacy v1 model); while the input is visible AND non-empty it wins on Apply, otherwise the picker selection is used. Typed value is RETAINED across toggles so the user can flip to the picker for a quick compare without losing work. Icon: `✏` enters edit, `↩` returns to picker. `Enter` blurs the input (commit-and-keep) so the user can tab into Apply without a mouse round-trip.
- **Per-row Play button** → opens the standalone `VideoPreviewModal` overlay for that file (does NOT commit any rename).
- **Confidence badge** → green for ≥ 0.6, amber for below.
- **Low-confidence row highlight** → yellow border + tinted background so the user can scan for "review me" rows at a glance.
- **Apply** → sequentially POSTs `/files/rename` for each checked row. Successful rows transition to a green "Renamed" state; failed rows stay visible with the error inline. The modal closes automatically when every applied row succeeds; otherwise it stays open so the user can see what failed.
- **Close** → atom cleared; the modal vanishes. Already-applied renames are not rolled back.

## Atom wiring

| Atom | Role |
|------|------|
| `smartMatchModalAtom` | `{ jobId, stepId, sourcePath, suggestions } \| null` — visibility + pre-ranked payload |
| `videoPreviewModalAtom` | Set by the per-row Play button to open the standalone preview |

## Scoring

Worker 25 relocated the scorer to `packages/core/src/app-commands/nameSpecialFeaturesDvdCompareTmdb.rankCandidates.ts` (server-side). The modal reads `rankedCandidates: ScoredCandidate[]` straight from the SSE payload — no client-side ranking pass. The web side keeps `LOW_CONFIDENCE_THRESHOLD = 0.6` in `smartMatchTypes.ts` purely for the UI threshold check (the server uses the same constant).

## Server endpoint

`POST /files/rename` body `{ oldPath, newPath }` — single-file rename. The route at `packages/api/src/api/routes/fileRoutes.ts:359` validates both paths for absolute / no-traversal safety and aborts if `newPath` already exists (so the user can't silently overwrite by misclicking).
