# RenameRegexField

An editor for the `renameRegex` field used by `copyFiles`, `moveFiles`, and `renameFiles`. Worker 65 extended the original two-input editor with a `flags` knob, an optional `sample` live-tester, and a slash-form display toggle. Worker 6e broadened the field to accept an **ordered chain of rules** applied left-to-right — each rule's output becomes the next rule's input.

## Behaviour

- **Single rule (default)** — starts as a single `{ pattern, replacement, flags?, sample? }` editor. Existing templates load unchanged.
- **Add rule** — clicking **+ Add rule** converts the field to chain mode and appends a blank second rule. The first rule's data is preserved.
- **Chain mode** — each rule row shows a Rule N label, ↑/↓ reorder buttons, and a ✕ delete button. Deleting down to one rule keeps the array form on the wire.
- **Per-rule live preview** — each row's "Test against" field runs `new RegExp(pattern, flags).exec(sample)` and shows a Match / No match badge with the predicted output and capture groups.
- **Chain output** — when any rule has a sample, a "Chain output" row at the bottom shows the result of applying every rule left-to-right to that sample. This makes "rule 3 broke it" visible at a glance.
- **Write — minimal wire shape** — single-rule mode emits the legacy 2-key `{ pattern, replacement }` when `flags` and `sample` are both empty. With either set it promotes to the 4-key object. When every field is blank it writes `undefined`. Chain mode always emits an array.
- **Slash-form toggle** — "Show as /…/" flips presentation between separate pattern + flags inputs and a single `/pattern/flags` literal. The underlying schema stays the separated pair.
- **Flag validation** — unknown flag chars mark the input `aria-invalid` with an inline tooltip.
