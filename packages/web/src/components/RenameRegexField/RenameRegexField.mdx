# RenameRegexField

A four-input editor (pattern + flags + replacement + sample) for the nested-object `renameRegex` field used by the `copyFiles` and `moveFiles` commands. The server applies the pair as `name.replace(new RegExp(pattern, flags), replacement)` against each copied/moved entry's basename. Worker 65 extended worker 63's two-input editor with the `flags` knob, an optional `sample` field that drives a client-side live tester, and a slash-form display toggle for users who prefer `/pattern/flags`.

## Behaviour

- **Read** — `step.params.renameRegex` is interpreted as `{ pattern, replacement, flags?, sample? }`. Missing keys collapse to `""`.
- **Write — minimal wire shape** — when `flags` and `sample` are both empty the component emits the legacy 2-key `{ pattern, replacement }` so YAML diffs stay small for users who never touch the new fields. With either set it promotes to the 4-key object. When *every* field is blank it writes `undefined` so `buildParams` strips the key entirely.
- **Live tester** — typing into "Test against (optional)" runs `new RegExp(pattern, flags).exec(sample)` and shows a Match / No match badge plus the predicted output filename and any captured groups. Pure client-side; no server round-trip.
- **Slash-form toggle** — "Show as /…/" flips presentation between the separate pattern + flags inputs and a single `/pattern/flags` JS-regex-literal input. The underlying schema stays the separated pair.
- **Flag validation** — typing an unknown flag char marks the input `aria-invalid` and surfaces an inline tooltip listing the offending chars (no exceptions).

## Props

| Prop | Type | Notes |
|------|------|-------|
| `step` | `Step` | Source of `params.renameRegex` and `command` (passed to `FieldLabel`) |
| `field` | `CommandField` | `{ name: "renameRegex", type: "renameRegex", label?, isRequired? }` |
